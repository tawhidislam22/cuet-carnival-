1   Project Overview

1.1   Project Title

CUET Carnival: Campus Event Management System

1.2   Problem Statement

CUET regularly hosts academic, technical, social, and cultural events involving hundreds of students. Historically, event registration has been handled through social media posts, paper forms, and informal messaging groups — leading to missed deadlines, duplicate registrations, lost records, and no centralized visibility for organisers or students. CUET Carnival addresses this by providing a unified, role-aware web platform where students can discover and register for events, organizers can publish and manage them, and administrators can oversee the entire ecosystem through a dedicated dashboard.

1.3   Objectives

  • Build a CUET-specific, centralized event discovery and registration platform.
  • Provide role-based workflows for students, organizers, and administrators.
  • Implement secure, institution-restricted authentication using CUET student email addresses.
  • Enable digital certificate issuance and revocation for event participants.
  • Design a modular, extensible REST backend that can be deployed on Vercel (serverless).

1.4   Target Users

  • CUET Students: Discover published events and register with a single click.
  • Organizers: Create events, upload banners, manage registrations, and issue certificates.
  • Administrators: Full platform control — publish/unpublish events, manage users, view reports, configure settings.

2   System Architecture

2.1   Architecture Overview

CUET Carnival follows a 3-tier client-server architecture with a strict separation of concerns:

  • Presentation Layer: Next.js 16 (App Router) frontend consumed by all user roles.
  • Application Layer: Node.js + Express REST API enforcing business rules, validation, and authorization.
  • Data Layer: Neon Postgres (serverless PostgreSQL) accessed via Prisma ORM.

Session management is delegated to Better Auth, which issues HTTP-only session cookies. Static event banner images are stored in a /uploads directory served as static files locally and in /tmp/uploads on Vercel.

3   Backend Design

3.1   Module Breakdown

The backend follows a feature-based modular pattern inside src/modules/. Each module owns its route, controller, service, and schema files:

  • auth — Delegates entirely to Better Auth. Handles sign-up (CUET email enforced), sign-in, sign-out, email verification, and password reset. Sessions are issued as HTTP-only cookies.
  • users — Exposes the authenticated user's profile (GET /me), organizer onboarding (PATCH /me/onboarding), and a list of organizers for the event creation form.
  • events — Full lifecycle: create, read, update, delete, image upload, registration, certificate issue/revoke. Owner-only writes enforced at service level.
  • dashboard — Aggregated views for users (overview, my-events, schedule, certificates) and admin-only endpoints (platform stats, user/event management, reports, settings).
  • admin — Reference data management (faculties, categories, venues) used to populate dropdowns in the create-event form. Admin-protected writes; public reads.

A shared requireAuth middleware extracts the session from the cookie, validates it against the database via Better Auth, and attaches req.authUserId and req.authUserRole to every authenticated request.

3.2   Workflow Description

  1. A CUET student signs up using their @student.cuet.ac.bd email; a verification link is sent via Gmail SMTP.
  2. After email verification, the student logs in and receives an HTTP-only session cookie.
  3. The student browses published events on the home page and clicks Register.
  4. The backend validates the session, checks capacity, and prevents duplicate registrations before creating an EventRegistration record.
  5. An organizer creates an event via the dashboard form; the event is saved as isPublished: false until an admin publishes it.
  6. An admin reviews events, toggles publish status, and can manage registrations or issue participation certificates.
  7. The student can view their registered events, upcoming schedule, and earned certificates from the personal dashboard.

4   Database Design

4.1   Database Type

The project uses PostgreSQL hosted on Neon (serverless). Prisma ORM provides type-safe query building and handles migrations. The serverless nature of Neon means the database may experience cold-start latency, handled gracefully by the backend with a 503 response on Prisma connection errors (P1001 / P1002).

5   API Design

5.1   API Overview

CUET Carnival backend exposes a RESTful JSON API. Endpoints are grouped by resource (/api/auth, /api/users, /api/events, /api/dashboard, /api/admin). Standard HTTP methods are used: GET for reads, POST for creation, PATCH for partial updates, DELETE for removal. All protected routes require a valid Better Auth session cookie; admin routes additionally enforce role = "admin".

6   Authentication and Security

6.1   Authentication Method

The backend uses Better Auth — a modern, framework-agnostic authentication library. After a successful sign-in, Better Auth issues an HTTP-only session cookie (not a client-readable JWT). The cookie is validated on every protected request via the requireAuth middleware, which calls Better Auth's session introspection and attaches req.authUserId and req.authUserRole to the request object.

Session records are persisted in the Session table and expire at a configurable TTL. Signing out deletes the session from the database and clears the cookie.

6.2   Security Measures

  • Institutional email restriction: Sign-up is gated by a regex pattern (/^u\d+@student\.cuet\.ac\.bd$/i); non-CUET addresses are rejected with a descriptive error.
  • Email verification: A verification link is sent via Gmail SMTP before the account is usable. Unverified accounts cannot access protected routes.
  • Password hashing: Passwords are hashed by Better Auth (bcrypt internally) before being stored; plain-text passwords never reach the database.
  • HTTP-only cookies: Session tokens cannot be read by JavaScript, eliminating XSS-based token theft.
  • Role-based access control: Admin-only endpoints verify req.authUserRole === "admin" before executing; non-admin calls receive 403 Forbidden.
  • Owner-only writes: Event update and delete verify that the requesting user is the organizerId of the event; mismatches return 403.
  • Input validation: Request bodies are parsed with Zod schemas (createEventSchema, updateEventSchema, registerForEventSchema) before reaching the service layer; validation failures return 400 with structured error details.
  • CORS: Cross-Origin Resource Sharing is restricted to the CLIENT_ORIGIN environment variable with credentials allowed.
  • Error sanitisation: The global error handler avoids leaking internal stack traces; Prisma-specific codes are mapped to user-friendly messages.

7   Error Handling

The backend uses a centralised error-handling strategy provided by express-async-errors, which automatically forwards unhandled async exceptions to the global error middleware without requiring per-route try/catch blocks.

8   Testing

8.1   Testing Strategy

API endpoints were tested manually using Thunder Client (VS Code extension) and curl throughout development. Each module was verified against its expected happy-path and error-path behaviour. SMTP configuration was validated using a dedicated npm run smtp:check script before deployment.

  • Authentication flow — sign-up, email verification, sign-in, sign-out, forgot/reset password.
  • Event CRUD — create, read, update, delete, image upload, publish toggle.
  • Registration flow — register, duplicate prevention, capacity enforcement, certificate issue/revoke.
  • Role-based access — admin-only routes blocked for regular users; owner-only writes enforced.
  • Error cases — invalid input, missing session, non-existent resources.

9   Limitations and Future Work

9.1   Limitations

  • Event banner images are stored on the local filesystem; on Vercel they are written to /tmp which is ephemeral — images are lost on function cold starts. A dedicated object storage (S3 / Cloudinary) is needed for production.
  • Certificate issuance currently creates a database record only; no PDF or QR-code generation is implemented.
  • No real-time notifications (WebSocket / Server-Sent Events) — status changes require a page refresh.
  • Search and filtering on event listings are limited; full-text search is not yet supported.
  • No automated test suite (unit or integration tests); testing was done entirely manually.
  • Neon free-tier cold starts can add 1–3 seconds of latency to the first request after idle.

9.2   Future Improvements

  • Integrate cloud storage (AWS S3 or Cloudinary) for permanent event image hosting.
  • Generate downloadable PDF participation certificates with QR code verification.
  • Add real-time email and in-app notifications for registration confirmations and status updates.
  • Implement full-text search and facet filtering on events (category, date range, capacity).
  • Write a comprehensive test suite using Vitest or Jest with database mocking.
  • Add a department-wise and event-wise analytics dashboard for administrators.
  • Extend authentication with OAuth (Google SSO) as a complement to email/password login.
  • Develop a mobile application (React Native or Flutter) consuming the same REST API.

10   Conclusion

CUET Carnival provides a structured and practical backend solution for managing campus events at Chittagong University of Engineering & Technology. It digitalises the full lifecycle of an event — from creation by an organizer through administrative review to student registration and certificate issuance — using a clean RESTful API, secure session-based authentication, and a relational PostgreSQL database.

The modular Express architecture, type-safe Prisma ORM, and Zod-based validation make the codebase maintainable and straightforward to extend. Role-based access control, institutional email restrictions, and HTTP-only cookies ensure that the platform meets fundamental security requirements.

As a result, manual overhead associated with spreadsheet-based registration is eliminated, and coordination between students, organizers, and administrators becomes transparent, trackable, and efficient. The current architecture serves as a solid foundation for the future improvements outlined above.
